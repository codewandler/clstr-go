# Design: Fix EventStore.Load Premature Termination (§2.3)

**Date**: 2026-04-08  
**Scope**: `adapters/nats/store.go` — `consumeEvents` and `EventStore.Load`  
**Review reference**: `review_reviewed.md` §2.3 — "Event Loading Doesn't Support Pagination"  
**Severity**: Production incident — two accounts (`610822ef`, `124478c1`) stuck in infinite retry loop in `latest` namespace

---

## Problem Statement

`EventStore.Load` uses an NATS ordered consumer with `FetchNoWait(100)` to retrieve events
for an aggregate. `FetchNoWait` returns only what is **immediately buffered** in the
server-side consumer delivery queue at the moment of the call. It does not wait for NATS
to scan ahead for more matching messages.

For a subject-filtered consumer on a large stream (MINI_ACD_ES: 2.8M messages,
`first_seq=8361`, `last_seq=2,877,900`), a subject with 1% message density has ~96
non-matching messages between consecutive events. After the first batch of 100 matching
events is consumed, the server-side queue empties while the server scans ahead. A
subsequent `FetchNoWait` call returns **zero messages** — `empty=true` — and the loop
exits, even though tens of thousands of events remain.

### Failure cascade (confirmed in production)

1. `repo.Load` receives only events 1–100 (partial). `curVersion=100` — no error, no
   `ErrAggregateNotFound`. Load appears to succeed.
2. `doSyncAccount` tries `Save` with `expectedVersion=100`; actual NATS version is 29,791.
   → **"concurrency conflict: expected version 100, got 29791"**
3. Save fails → no Redis snapshot is written.
4. On retry: ordered consumer occasionally auto-resets (its normal self-healing) and
   re-delivers from seq 1. `consumeEvents` appends re-delivered events to the same slice.
   `repo.Load` sees `[v1..v100, v1...]` → **"expect version 101, got 1"**
5. After `Load` errors, `GetOrCreate` falls through to `Create` → tries to append version
   1 to a stream at version 29,791 → **"expected version 0, got 29791"**
6. **Infinite loop.** These accounts have never successfully snapshotted since the Redis
   migration (2026-04-03, v0.32.0).

---

## Root Cause

```go
// adapters/nats/store.go — consumeEvents
mb, err = cc.FetchNoWait(100)  // ← exits on empty; does not wait for server scan
...
if empty {
    break  // ← premature exit for sparse subject filters on large streams
}
```

The `endSeq` termination condition (correct in design) is never reached because the loop
exits first via `empty`.

Secondary: `startVersion` from `WithStartAtVersion` is captured in `storeLoadOptions` but
**never applied** to the consumer config. The NATS consumer has no version concept, so
this is correct by design — but the variable is dead code and should be removed to avoid
confusion.

---

## Solution

### Primary fix: replace `FetchNoWait` with `Fetch` + max-wait timeout

`Fetch(n, jetstream.FetchMaxWait(d))` blocks up to `d` for the server to return `n`
messages. This gives NATS time to scan through non-matching messages in the stream and
fill the delivery queue, eliminating the premature exit.

The loop exits cleanly when:
- `ev.Seq >= endSeq` (correct path: all events loaded), **or**
- `Fetch` times out returning zero messages (NATS genuinely has nothing — should not
  happen for existing events, only when the stream is truly exhausted)

**Timeout value**: `5 seconds`. Rationale: NATS subject-filter scanning is CPU-bound and
completes in microseconds under normal load. 5 s gives large headroom for JetStream under
peak pressure without degrading load latency meaningfully (events are processed in-memory
once received; the bottleneck is network round-trips, not CPU).

The timeout is exposed as `EventStoreConfig.LoadFetchTimeout` with a default of `5s` so
callers can tune it if needed without a library change.

### Secondary fix: remove dead `startVersion` variable

`startVersion` is set from `loadOpts.startVersion` but is only used in the `slog.Debug`
call. Remove the local variable; log `loadOpts.startVersion` directly. This makes the
code honest about what is and isn't enforced.

### Out of scope

- Version-based filtering inside `consumeEvents`: the `repo.Load` version-ordering check
  is the correct enforcement point. Duplicating it in the store would be premature.
- Fixing `startVersion` enforcement at the store level: out of scope; the NATS consumer
  has no version concept and the `repo.Load` guard is sufficient.
- Ordered consumer reset detection: with `Fetch` + timeout the reset window is effectively
  eliminated because the loop no longer prematurely exits. The `repo.Load` version check
  catches any re-delivery if a reset does occur.

## Test Plan

### New test: `TestStore_Load_ManyEvents`

Reproduces the production failure:

1. Append **250 events** for aggregate A and **250 events for a second aggregate B**,
   interleaved (A, B, A, B, …) to simulate sparse subject density.
2. Call `EventStore.Load` for aggregate A.
3. Assert all 250 events for A are returned in version order.

This test fails with `FetchNoWait` (returns ~100 events) and passes with `Fetch` +
timeout.

### Regression: existing store tests

All existing tests in `adapters/nats/store_test.go` must continue to pass.
